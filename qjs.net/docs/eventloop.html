事件循环 & Promise

qjs.net 的运行时模型与 Promise 调度规则。

线程模型

  • 每个 QuickJSEngine 拥有一个 QuickJS Runtime + Context，单线程使用。
  • 所有 JS 代码（包括从 Task 转换来的 Promise 回调）都在调用 Execute() 的线程上执行。
  • C# async 方法在 JS 端 await 后回到 JS 时，回调会被 marshal 回 原 EventLoop 线程 —— 你不会在 JS 中看到跨线程 race。
  • 不要从多个 .NET 线程并发调用同一个 QuickJSEngine。

Execute 的语义

engine.Execute(code) 大致等价于：

var p = JS_Eval(code, JS_EVAL_FLAG_ASYNC);   // 包成 async 函数
while (!settled(p)) {
    JS_ExecutePendingJob();                       // 先解 microtask
    eventLoop.RunOneTick();                       // 再处理定时器/已完成 Task
}
return resolveValue(p);

因此 Execute 内可写：

await fetch('https://...');
await fsAsync.writeFile('log.txt', 'done\n');

Microtask & Promise

  • QuickJS 内部维护一个 microtask 队列；EventLoop 在每次 tick 都会先 drain 它。
  • queueMicrotask(fn) 全局可用。
  • 所有 .then(...) 链都在同一个 EventLoop 上推进；(await fetch(url).then(r => r.json())).x 会正确解开。

定时器

setTimeout(fn, delayMs) / setInterval(fn, periodMs) / clearTimeout / clearInterval 全部由内置 EventLoop 调度。它们只在 Execute 还在 pump 时触发——没有后台线程偷偷推进时钟。

如果你需要后台进程持续运行，可在 .NET 端开线程，反复调用 engine.Execute("eventLoop.runOnce()") 之类的入口（参见 EventLoop.RunUntilDone）。

Task ↔ Promise 互通

C# 端：JSPromiseBridge.FromTask(rt, task) 把 Task 转成 JS Promise；continuation 通过 EventLoop.Post() marshal 回主线程。

JS 端：所有 [JSExport] 方法若 C# 签名为 async，在 JS 中即返回 Promise；await 即可。

不要做的事

不要在被 JS 调用的同步 C# 方法里 .Wait() 或 .Result 一个未完成的 Task——若该 Task 的完成依赖 EventLoop 推进（例如又 post 了一个回调），会死锁。改为：把方法标 async，让生成器把它桥成 Promise。
